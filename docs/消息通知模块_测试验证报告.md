## 完整审批流程验证

### 场景：提交 → 通过 → 撤回

#### 步骤1：提交申请
```bash
curl -X POST "http://localhost:8080/approval/submit" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer {applicant_token}" \
  -d '{
    "approvalType": "leave",
    "title": "2026年7月请假",
    "content": "请假5天",
    "startTime": "2026-07-22T09:00:00",
    "endTime": "2026-07-27T17:00:00",
    "leaveType": "年假"
  }'
```

**数据库验证**：
```sql
-- 检查审批表
SELECT id, status, applicant_id FROM t_approval 
WHERE applicant_id = {applicant_id} AND status = '待审批'
ORDER BY create_time DESC LIMIT 1;

-- 检查消息表 - 应有SUBMITTED事件
SELECT 
  id, event_type, is_todo, is_read, biz_id 
FROM t_message 
WHERE event_type = 'SUBMITTED' AND biz_id = {approval_id}
LIMIT 1;
```

**预期结果**：
- ✅ `t_approval` 新增1条记录，status='待审批'
- ✅ `t_message` 新增1条记录，event_type='SUBMITTED', is_todo=1, is_read=0
- ✅ message.jump_url = '/approval/detail/{approval_id}'

#### 步骤2：审批人查看待办消息
```bash
curl -X GET "http://localhost:8080/message/list?category=todo&page=1&size=10" \
  -H "Authorization: Bearer {approver_token}"
```

**预期结果**：
- ✅ 返回待办消息列表，包含刚才提交的SUBMITTED消息
- ✅ is_todo=1, is_read=0

#### 步骤3：审批人标记消息已读并跳转
```bash
curl -X PUT "http://localhost:8080/message/read/{message_id}" \
  -H "Authorization: Bearer {approver_token}"
```

**预期结果**：
- ✅ 返回jumpUrl="/approval/detail/{approval_id}"
- ✅ `t_message.is_read` 变为1

#### 步骤4：审批人通过申请
```bash
curl -X PUT "http://localhost:8080/approval/approve" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer {approver_token}" \
  -d '{
    "id": {approval_id},
    "status": "已通过",
    "approveReason": ""
  }'
```

**数据库验证**：
```sql
-- 检查审批记录已更新
SELECT status, approver_id FROM t_approval 
WHERE id = {approval_id};

-- 检查消息表 - 应有APPROVED_FINAL事件
SELECT 
  id, event_type, receiver_id, is_todo, is_read, sender_name
FROM t_message 
WHERE event_type = 'APPROVED_FINAL' AND biz_id = {approval_id}
LIMIT 1;
```

**预期结果**：
- ✅ `t_approval.status` = '已通过'
- ✅ `t_approval.approver_id` = 审批人ID
- ✅ `t_message` 新增1条记录，event_type='APPROVED_FINAL', is_todo=0, is_read=0
- ✅ receiver_id 为申请人ID

#### 步骤5：申请人查看通过消息
```bash
curl -X GET "http://localhost:8080/message/list?category=all&page=1&size=10" \
  -H "Authorization: Bearer {applicant_token}"
```

**预期结果**：
- ✅ 返回消息列表，包含APPROVED_FINAL消息
- ✅ 内容包含"您的XX申请已审批通过"

#### 步骤6（拒绝场景）：拒绝申请
```bash
curl -X PUT "http://localhost:8080/approval/approve" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer {approver_token}" \
  -d '{
    "id": {approval_id},
    "status": "已拒绝",
    "approveReason": "请假时间过长"
  }'
```

**数据库验证**：
```sql

  id, event_type, content, is_todo
FROM t_message 
WHERE event_type = 'REJECTED' AND biz_id = {approval_id}
LIMIT 1;
```

**预期结果**：
- ✅ `t_message` 新增1条记录，event_type='REJECTED', is_todo=0
- ✅ content 包含拒绝原因："请假时间过长"

#### 步骤7：撤回申请（WITHDRAWN事件）
```bash

  -H "Authorization: Bearer {applicant_token}"
```

- ✅ `t_approval.status` = '已撤回'

## 事件处理链路验证

### 事件发布 → 监听 → 消息生成

1. **事件发布点**（ApprovalServiceImpl）
   - ✅ submitApproval: 发布SUBMITTED
   - ✅ approveApproval: 发布APPROVED_FINAL/REJECTED
   - ✅ withdrawApproval: 发布WITHDRAWN

2. **事件监听**（ApprovalEventListener）
   - ✅ @EventListener 方法监听ApprovalEvent
   - ✅ 根据eventType生成对应消息
   - ✅ 异常处理不影响主业务流程

3. **消息存储**（MessageService）
   - ✅ sendMessage 将消息持久化到t_message
   - ✅ 字段填充完整（event_type, biz_type, is_todo等）

## 故障排查指南

### 问题1：消息未生成
**现象**：提交申请后，t_message表无新记录
**排查步骤**：
1. 检查应用日志中ApprovalEventListener是否收到事件
2. 验证MessageService.sendMessage是否被调用
3. 检查消息表字段是否全部已添加

**解决方案**：
```sql
-- 确认字段存在
DESCRIBE t_message LIKE 'event_type';
DESCRIBE t_message LIKE 'is_todo';

-- 手动测试消息插入
INSERT INTO t_message (receiver_id, sender_id, title, content, event_type, is_todo)
VALUES (2, 1, 'Test', 'Test message', 'SUBMITTED', 1);
```

### 问题2：待办消息查询为空
**现象**：/message/list?category=todo 返回空列表
**排查步骤**：
1. 检查is_todo字段是否设置正确
2. 验证is_read字段是否为0
3. 确认当前登录用户ID与receiver_id一致

**解决方案**：
```sql
-- 验证待办消息
SELECT COUNT(*) FROM t_message 
WHERE receiver_id = {current_user_id} 
AND is_todo = 1 
AND is_read = 0;
```

### 问题3：越权访问未被阻止
**现象**：用户能查看别人的消息
**排查步骤**：
1. 检查MessageController中是否验证receiver_id
2. 确认UserContext.getCurrentUser()返回正确用户信息

**解决方案**：
```sql
-- 检查消息访问日志
SELECT DISTINCT receiver_id FROM t_message 
WHERE id = {accessed_message_id};
```
