## 🏗️ 架构设计

### 解耦流程图

```
┌─────────────────┐
│ ApprovalController
│  (审批控制器)      │
└────────┬─────────┘
         │
         ▼
┌─────────────────────────┐
│ ApprovalServiceImpl       │
│  (审批业务逻辑)         │
│  - submitApproval()     │
│  - approveApproval()    │
│  - withdrawApproval()   │
└────────┬────────────────┘
         │ 发布事件
         │ publishEvent(ApprovalEvent)
         ▼
┌─────────────────────────┐
│ Spring ApplicationEvent │
│ Publisher (事件总线)    │
└────────┬────────────────┘
         │ 异步监听
         ▼
┌──────────────────────────┐
│ ApprovalEventListener    │
│ @EventListener           │
│ (消息模块不感知业务规则)  │
└────────┬─────────────────┘
         │ 调用
         ▼
┌──────────────────────┐
│ MessageService       │
│ (消息持久化)         │
└────────┬─────────────┘
         │ 保存
         ▼
┌──────────────────────┐
│ t_message 表         │
│ (消息存储)           │
└──────────────────────┘
```

**解耦优势**：
- 审批模块只关心审批流程，不关心消息生成
- 消息模块只订阅事件，不感知审批业务规则
- 易于扩展：新的业务模块只需发布事件，消息模块自动处理

## 📡 API 接口汇总

### 消息中心接口

| 方法 | 路径 | 功能 | 权限 |
|------|------|------|------|
| GET | `/message/list` | 分页查询消息（支持分类筛选） | 登录用户 |
| GET | `/message/todoCount` | 获取待办数量 | 登录用户 |
| PUT | `/message/read/{messageId}` | 标记单条已读 | 登录用户 |
| PUT | `/message/readAll` | 批量标记已读 | 登录用户 |

### 分类参数说明
- `all` - 全部消息
- `todo` - 待办(is_todo=1且is_read=0)
- `cc` - 抄送(biz_type != 'approval')
- `system` - 系统通知(msg_type != 'APPROVAL')

## 🔄 完整流程示例

### 提交 → 通过 → 撤回

```
① 申请人提交
   POST /approval/submit
   ↓
   事件: SUBMITTED (receiver=审批人, is_todo=1)
   
② 审批人查看待办
   GET /message/list?category=todo
   ↓
   返回：待办消息列表
   
③ 审批人标记已读
   PUT /message/read/{messageId}
   ↓
   返回：jump_url = /approval/detail/{id}
   
④ 审批人通过申请
   PUT /approval/approve
   ↓
   事件: APPROVED_FINAL (receiver=申请人, is_todo=0)
   
⑤ 申请人查看审批结果
   GET /message/list?category=all
   ↓
   返回：通过消息
```

## 📞 技术支持

### 常见问题

**Q: 消息为什么没有生成？**  
A: 检查：
1. 应用日志中是否看到ApprovalEventListener的日志
2. t_message表中是否有新的记录
3. 事件监听器是否被正确注册（@Component注解）

**Q: 为什么看不到其他人的消息？**  
A: 这是正常的越权防护。消息系统会验证receiver_id是否等于当前用户，不相等则拒绝访问。

**Q: SUBMITTED事件为什么发给了系统管理员？**  
A: 当前版本是临时处理方案。等待工作流引擎完成后，会改为发送给实际的第一级审批人。
